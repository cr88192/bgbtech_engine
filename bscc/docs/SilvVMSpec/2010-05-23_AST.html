This spec will attempt to document the ASTs used in my compiler front-end.

General conventions:
<nodetype/>
Will identify a particular XML node type.
This will refer to the node, rather than necessarily define its contents as empty.

[ stuff ]
Will indicate that "stuff" is optional.

something+
Will indicate that something may appear 1 or more times.

something*
Will indicate that something may appear 0 or more times.

If a single node type is defined multiple times, it means that it may appear in one of these forms.


Toplevel:

<module lang="srclang" arch="arch">

[<structs/>]
[<types/>]
<body/>

</module>


The lang attribute will give the name of the source language, which may effect the treatment of particular nodes, types, or modifier flags.

The arch attribute will identify the particular target architecture.
This is relevant mostly for languages like C and C++ where the code is typically specialized for a particular arch as early as the preprocessor.
The arch name "default" or "any" will identify that the code in question doesn't itself depend on any particular target arch.
Note that the target architecture may in-fact represent a request by the user for a particular target, rather than simply as a limitation of the source language.

<structs/>
This node will contain a list of known structures/unions/classes/etc...
This will be used in a language such as C or C++ where structures may be defined inline and subsequently referenced via a similar definition, and so provides a space in which to merge multiple such definitions.

There is no such requirement, however, that structured types actually be placed here (rather than, for example, in context under the body node), and so as a result this node may be empty or absent despite structured types having been declared.


<types/>
This node will contain a list of known type definitions.
As with <structs/>, this node is not necessarily used for type-definitions.
In general, types will be defined via <var/> nodes.


<body/>

Will contain all the "stuff" for the module, starting from the POV of the toplevel.


General Notes

The AST doesn't solidly distinguish between particular syntactic roles (statement vs expression, or expression vs declaration), and is generally a little looser than the source language. Instead, things are roughly organized by "context", and the particular types of nodes which may appear in a given context.


Statement context is generally where no return value would make sense from an expression.
Declarations are generally only allowed to appear in statement contexts.
Any expression may appear in a statement context, but an expression is considered as "useless" if it does not produce any side-effects.

Expression context is where a return-value is expected.
Statements which may not produce a meaningful return value may not appear in expression context (note that in many cases it could be possible to "reinterpret" these statements in a form where a return-value makes sense, but since these semantics can't be represented in the typical source languages, this usage doesn't make much sense).
Declarations are similarly disallowed as they produce no meaningful return value (and also due to source language).

Declaration context is where only declarations are allowed (as neither expressions nor statements would be sensible).

Type-Expression context overlaps with Expression context, but differs mostly in that operations are being applied to abstract types rather than to values, and as a result only certain expressions make sense, and there may be additional expression types which are only sensible in the context of types. Note that many operations may have different semantics when applied to types.


A subset of expression context is tail-expression context, which exists only when the current expression is part of a return (either explicitly via a return, or implicitly via being in tail-position within an expression already in tail position). Tail context does not exist within non-tail positions within an expression, even if the parent expression is within tail position. Generally, if any operation needs to be performed on the result of an expression prior to returning it, it is no longer in tail position (although, an exception may be made for casts where no type-conversion is involved).


Some nodes serve as lists, where lists are expected to have items in their regular ordering, and are as-such order and position sensitive. Any node present within a list is assumed to be a member of that list.

Most other nodes have their contents identified via markers. Marker nodes need not appear in any particular order, and generally (within reason) custom nodes may be placed within such a node without causing issue. Generally, marker nodes are also list nodes, and very often a marker node may only validly contain a single child node (this being the marked item).

Marker nodes generally serve the purpose of identifying a particular node as serving a particular role, but this node may be one of any number of types.


Exact numeric flags values are undefined here (flag strings would be better, albeit they are not presently used).
The flags-string format is specified in the metadata spec.

xmlns (if needed): "uuid:dab16484-68e9-4b0f-a6cc-883ff7cd0f00"
This will be applied to the toplevel module.

The ln and fn attributes may be applied to any general-purpose node type, and will indicate the source line number and filename.



Types and Definitions


Example Syntax

XML

Comments

modifiers-typenames-and-attributes1

<type name="name" flags="flags" ind="ind">
    [<size/>]
    [<rsize/>]
    [<attributes/>]
</type>

This identifies a unique type, which may later be referenced by expressions or declarations. Types will be considered equivalent if they have the same structure.


The flags attribute gives the modifier flags, and ind gives the pointer indirection level (C/C++ only).
The <size/> and <rsize/> nodes give the array size for C-style arrays.

1 C/C++ specific.

modifiers-and-attributes typename

<type name="name" flags="flags">
    [<attributes/>]
</type>

The name attribute identifies the specific type-name, which may be a built-in or externally-defined type, a structural definition, or a type-definition. This attribute may be unused if a type-expression defines the type.

modifiers-and-attributes type-expression

<type flags="flags">
    [<expr/>]
    [<attributes/>]
</type>

The <expr/> node may be used to define a type via a type-expression.

modifiers struct name body

modifiers struct attributes name body

struct body

<struct name="name" flags="flags">
    <body/>
    [<attributes/>]
    [<sharpargs/>]
</struct>

The name is required for structural types, even if none is given in the source language. In the case of anonymous structural types, a name is simulated, such as via the use of a hash function or gensym (less preferable as this can't as easily merge duplicate structs and may pollute external metadata with stale redundant declarations).

modifiers union name body

modifiers union attributes name body
union body

<union name="name" flags="flags">
    <body/>
    [<attributes/>]
    [<sharpargs/>]
</union>


modifiers enum name body

modifiers enum attributes name body
enum body

<enum name="name" flags="flags">
    <body/>
    [<attributes/>]
    [<sharpargs/>]
</enum>

Where name gives the particular structure name, and flags gives the modifier flags.
The body node gives the collection of definitions for the structure.

modifiers class name body

modifiers class attributes name:super,interfaces body

modifiers class attributes name<sharpargs> extends super implements interfaces body

class body

<class name="name" flags="flags">
    <super/>
    <impl/>
    <body/>
    [<attributes/>]
    [<sharpargs/>]
</class>

The super and impl nodes give the superclass and implemented interfaces for a class.
Both super and impl contain <type/> nodes.

type name(args);

<proto name="name"> <type/> <args/> </proto>

This gives function prototypes and method declarations.

type (name)(args);1
delegate type name(args);

<s_proto name="name">
    <type/>
    <args/>
</s_proto>

This structure-prototypes.
The main difference here is that a normal prototype works as a function declaration, whereas a structure-prototype behaves like a structure and may be used in declaring function-pointer types.

1 This is the type-of the declaration, but the declaration will itself produce different syntax.

type name { get set }

<property name="name">
    <type/>
    [ <get/> ]
    [ <set/> ]
</property>


type name
type name=value

<var name="name">
    <type/>
    [ <value/> ]
</var>

Declares a variable or field.
Not usually standalone except in an arguments list or similar.
Normally, each <var/> is a member in a <vars/> node.
